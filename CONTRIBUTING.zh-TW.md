# 貢獻指南

**[English version](CONTRIBUTING.md)**

感謝你的興趣！spec-sonar 是方法論專案，不只是程式碼專案——最有價值的貢獻是**蒸餾過的經驗**，不是功能。

## 我們最需要的

1. **新的暗區維度**——一個維度要進 `dark-zone-baseline.md`，你必須能指出一個真實專案因為沒人問這個問題而失敗（或大幅返工）。PR 描述請附上事後檢討的證據。
2. **產品類型擴充包**——目前涵蓋 Marketplace / B2B SaaS / 消費型 App / 內部工具 / ERP。新類型（如嵌入式、資料管線、遊戲）遵循同樣格式：2–3 個維度，每個附具體問題與亮區條件。
3. **Adapter 格式**——新目標平台（Windsurf、Aider、OpenHands…）。遵循 `docs/adapter-formats.md`：先說明該平台的**讀取模型**（它執行任務、編輯程式碼、還是只做補全？），再從讀取模型推導格式。
4. **端到端範例**——一個真實想法經過引擎收斂、分解、（最好）執行的完整紀錄。隱私內容請遮蔽。
5. **翻譯**——SKILL.md 的英文原生版是路線圖第一項。

## 基本規則

- **可 merge 的段落**：SKILL.md 的修改必須維持「固定行為規則」風格（編號協議、表格、回覆模板），不用角色扮演語言。
- **證據導向**：每個依賴類型、維度、協議都必須附至少一個它能防止的具體失敗案例。
- **自包含原則神聖不可侵犯**：任何讓 goal 檔依賴外部脈絡的修改都會被拒——它會破壞冷啟動測試。
- **雙語文件**：README 層級的文件以英文與繁體中文出貨；深度設計文件可先以任一語言出貨，標記待翻譯。
- `tools/project-scanner.py` 必須維持**純標準庫**，輸出小於 20 KB。

## 工作流程

1. 大型 PR 前先開 issue 描述缺口。
2. 從 `main` 開分支，每個 PR 聚焦一件事。
3. Schema 變更（`STATE`、`goal-graph`）需升版號並更新 `CHANGELOG.md`。
