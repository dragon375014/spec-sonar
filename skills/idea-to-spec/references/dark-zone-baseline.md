# 暗區基準清單

<!-- spec-sonar v1.1 — 含五種產品類型擴充維度 -->

軟體產品的 11 個標準維度。用來初始化每個新 session 的 `dark` 清單。
根據用戶第一句話已透露的資訊，把已知維度移入 `bright`，其餘留在 `dark`。

---

## 維度清單

### 1. 目標用戶
- 誰是主要使用者？
- 有幾種角色？（例：會員、管理員、教練）
- 用戶的技術能力如何？
- **亮區條件**：明確說出至少一種用戶群體

### 2. 核心功能邊界
- 最核心的 1–3 個功能是什麼？
- 明確不做哪些功能？（防止範圍蔓延的關鍵）
- **亮區條件**：列出具體功能名稱，且有「不做」的確認

### 3. 資料模型
- 系統需要儲存哪些資料？
- 有沒有敏感資料？（個資、金流、健康資料）
- 資料的保存期限？
- **【關鍵分叉】Session 資料 vs 持久資料**：
  - 資料需要跨次保留嗎？（歷史記錄、用戶進度、累積數據）
  - 還是每次使用結束後資料可以消失？（課堂遊戲、一次性活動）
  - 這個分叉決定是否需要帳號系統，影響架構複雜度極大，必須主動問清楚
- **亮區條件**：確認主要資料實體，且明確 Session-only 或 Persistent

### 4. 認證與權限
- 需要登入嗎？
- 不同角色有不同的操作權限嗎？
- 登入方式？（帳密 / Google / LINE / 手機號碼）
- **亮區條件**：確認是否需要登入及角色設計

### 5. 第三方整合
- 需要串接金流嗎？（LINE Pay / 信用卡 / 轉帳）
- 需要串接通知嗎？（LINE Notify / Email / SMS / Push）
- 有現有系統需要對接嗎？（ERP / POS / 預約系統）
- **亮區條件**：確認整合清單（可以是空清單）

### 6. 部署平台
- Web 網頁？手機 App？桌面應用？
- 手機優先還是電腦優先？
- 需要上架 App Store / Google Play 嗎？
- **亮區條件**：確認主要使用裝置與平台

### 7. 效能與規模
- 預計同時使用人數？
- 資料量規模（小型 <1000 筆 / 中型 / 大型）？
- 需要即時性嗎？（即時聊天、直播等）
- **亮區條件**：有大概的規模認知

### 8. 維護與更新
- 誰來維護這個系統？
- 多久需要更新一次內容？
- 需要後台管理介面嗎？
- **亮區條件**：確認維護責任歸屬

### 9. 預算與時程
- 大概的預算範圍？（影響技術選型）
- 希望多久可以上線？
- 是一次性開發還是長期維護？
- **亮區條件**：有大致的限制條件

### 10. 成功指標
- 怎麼判斷這個系統「做好了」？
- 上線後最重要的一件事是什麼？
- 有沒有需要達到的業務目標？
- **亮區條件**：用戶能說出至少一個驗收標準

### 11. 用戶旅程與入口
- 每種角色打開瀏覽器 / App 後看到的**第一個畫面**是什麼？
- 從第一個畫面到完成核心任務，中間經過哪些畫面？每一步靠什麼前進（按鈕、連結、自動跳轉）？
- 多角色產品：不同角色在哪裡分流？（同一入口選身份，還是各自不同的 URL？）
- 跨角色的觸發鏈：一個角色的動作，另一個角色在什麼畫面看到什麼結果？
  （例：老師點「開始」→ 學生畫面收到題目——這類「邊」是模組式分解最容易弄丟的東西）
- **亮區條件**：每個角色都有一條從入口到核心任務完成的完整路徑描述

## 類型擴充維度

基準 11 維是所有軟體共用的。以下五種產品類型各有額外的高頻死角。
**載入邏輯**：步驟三初始化時，先從用戶第一句話判斷產品類型
（判斷不出 → 留到第 1 輪意圖擷取後補載入），把對應擴充維度併入 dark 清單，
優先級插在「第三方整合」之後、「效能規模」之前。

### 類型 A：Marketplace（雙邊平台）

**A-11. 雙邊冷啟動**
- 先有供給方還是需求方？最初 50 個供給從哪來？
- 供給不足時需求方看到什麼？（空市場是平台死因第一名）
- 亮區條件：說得出冷啟動的第一步具體做法

**A-12. 交易與抽成模型**
- 錢流經平台（託管/escrow）還是雙方直接交易？
- 誰付手續費？退款與糾紛誰裁決？
- 亮區條件：確認金流路徑與糾紛處理責任方

**A-13. 信任與評價機制**
- 雙向評價？供給方審核（KYC）？詐騙與跳單防範？
- 亮區條件：確認至少一種信任建立機制

### 類型 B：B2B SaaS（多租戶企業用）

**B-11. 租戶隔離模型**
- 共用資料表加 tenant_id，還是 schema-per-tenant，還是獨立部署？
- 客戶會不會要求「我的資料不能跟別人放一起」？
- 亮區條件：確認隔離等級與資料歸屬

**B-12. 企業身份與組織權限**
- 需要 SSO / SAML / Google Workspace 嗎？
- 組織內有層級嗎（admin 委派、部門隔離、審批鏈）？
- 亮區條件：確認登入方式與組織模型

**B-13. 計費與方案結構**
- per-seat / usage-based / 固定方案？試用期？發票與合約（企業採購流程）？
- 亮區條件：確認計費單位（哪怕是「第一版免費」也算）

### 類型 C：消費型 App（C 端社群/娛樂）

**C-11. 內容審核與安全**
- 有 UGC 嗎？審核是事前、事後還是檢舉制？
- 未成年用戶？封鎖/檢舉/隱私設定？（上架審查會問）
- 亮區條件：確認 UGC 有無與最低審核機制

**C-12. 留存與成長迴路**
- 用戶第二天為什麼回來？通知策略？邀請機制？
- 亮區條件：說得出至少一個留存 hook

**C-13. 變現模式**
- 廣告 / 訂閱 / IAP / 完全免費？（影響架構：廣告 SDK、收據驗證）
- 亮區條件：確認第一版的變現（含「不變現」）

### 類型 D：內部工具（公司內部非對外）

**D-11. 既有系統與身份整合**
- 公司有 AD / Google Workspace / SSO 嗎？工具要不要接？
- 資料來源是哪個既有系統（ERP/CRM/Excel）？單向同步還是雙向？
- 亮區條件：確認身份來源與資料上游

**D-12. 資料治理與稽核**
- 誰能看誰的資料（跨部門隔離）？需要操作紀錄（audit log）嗎？法遵要求？
- 亮區條件：確認可見性規則

**D-13. 導入與舊流程退役**
- 現在這件事怎麼做（Excel？紙本？）？舊資料要不要搬進來？
- 誰負責推動同事改用新工具？
- 亮區條件：確認遷移範圍與推動者

### 類型 E：ERP / 複雜企業系統

**E-11. As-is 流程盤點**
- 現行流程（含例外流程、私下的 Excel 流程）有沒有文件化？
- 哪些流程要照搬、哪些要趁機改造？（照搬爛流程是 ERP 失敗主因）
- 亮區條件：每個目標模組都有對應的現狀描述

**E-12. 主數據歸屬（Master Data）**
- 客戶、料號、會計科目表——每種主數據誰建立、誰修改、誰是唯一真相源？
- 模組間共用主數據的同步規則？
- 亮區條件：列出主數據清單與各自的 owner

**E-13. 遷移與並行上線**
- 舊系統資料要搬多少年份？髒資料誰清？
- 新舊系統並行多久？並行期間以哪邊為準？
- 亮區條件：確認遷移範圍與切換策略

### 對輪次骨架的影響

類型擴充維度啟用時，輪次上限從 7 放寬到 9（Complex System 模式），
handoff 門檻同步調整為 `round >= 6 且 dark <= 3`（ERP 類允許帶著
較多已知未知進入分解，由 goal-decomposer 的 Audit 迴圈接手）。

## 初始化邏輯

```
步驟 0：判斷產品類型
  （Marketplace / B2B SaaS / 消費型 App / 內部工具 / ERP / 一般）
  → 命中類型 → 將對應擴充維度併入 dark 清單

for each 維度 in (11 個基準維度 + 已載入的類型擴充維度):
  if 用戶第一句話已涵蓋此維度:
    加入 bright
  else:
    加入 dark

dark 清單按重要性排序：
優先級高：目標用戶 > 功能邊界 > 用戶旅程與入口 > 認證權限 > 第三方整合 > [類型擴充維度]
優先級低：效能規模 > 維護更新 > 預算時程 > 成功指標
```

`next_focus` 每輪取 `dark` 清單中優先級最高的未確認項目。
