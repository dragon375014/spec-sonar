## M0. modality 分叉樁（最上游 — 先於 11 維與產品類型問）

<!-- G13：補「整套樁預設螢幕」盲區。modality 選錯不會炸在下游契約/BLOCKED，只在上線後客戶抱怨才現身=最貴偵測點 → 必須收斂期最上游問。 -->

下面 11 維 + 5 類型包**預設「產品 = 有畫面的 Web/App」**。但「主要介面是不是螢幕」決定**整套樁問不問得對方向**——比 a11y / i18n 更上游。**步驟三初始化前先問這一個分叉。**

**問什麼**：使用者主要**怎麼跟產品互動**？看畫面點按（網頁 / App）、用講的（語音助手 / 電話 IVR / LINE bot 對話）、程式對接（純 API / SDK，沒畫面）、收訊息（email / SMS / 推播為主）、還是拿實體輸出（列印標籤 / PDF / 報表）？

**亮區條件**：確認**唯一或主要** modality ∈ {螢幕 / 語音 / API / 訊息 / 列印}（多 modality 時確認哪個是主）。

| 答案 | 後續 |
|---|---|
| **螢幕**（Web/App/桌面） | 照常載 11 維 + 5 類型包（+ 螢幕型 UX 樁，若已安裝）|
| **語音 / 對話式**（chatbot/IVR/LINE bot）| ⚠️ 螢幕型 UX 假設不適用（改問多輪降級 / 誤解處理 / 無視覺確認）。**對應樁包尚未建** → 寫 `known_risks`「modality=語音，UX 樁待長」+ 旗標 user |
| **純 API / SDK**（無畫面）| ⚠️ UX = DX（文件 / 錯誤碼 / 版本相容 / rate limit）。**樁包未建** → known_risks + 旗標 |
| **訊息為主**（email/SMS/push）| ⚠️ 介面 = 文案 + 模板 + 寄送時機，不是頁面。**樁包未建** → known_risks + 旗標 |
| **列印 / 實體輸出**（ESC/POS、PDF）| ⚠️ 介面 = 排版 / 字寬 / 裁切 / 紙張。**樁包未建** → known_risks + 旗標 |

> **為什麼最上游、即使本版沒對應樁包也要先問**：modality 選錯**不產生下游機械訊號**（不是欄位缺、不是 trace 斷），只在**上線後**「這該是 LINE bot 不是網頁」才爆——逃掉後唯一偵測點、最貴。先問 = 把它從「靜默假設螢幕」變成「顯式分叉 + 已知缺口」；即使非螢幕樁包還沒長，也不再靜默走錯方向。

## M0.5. archetype 分叉樁（modality=螢幕 之下 — 先於載入類型包）

<!-- ADR-002：補「靜默假設交易型應用」盲區。archetype 選錯同 modality——不產生下游機械訊號，只在上線後客戶抱怨「後端 90 前端 10」才現身。 -->

modality=螢幕 後，再分**交付物原型（archetype）**——它決定載哪套類型包與 baseline。**步驟三載入類型包前先判這一個分叉**（同 M0：多數一次中，但仍顯式確認，因為猜錯＝上線才爆）。

**問什麼**：這是哪一種螢幕型東西？——做生意的系統（要交易/後台/多角色）、呈現自己的網站（品牌/作品集/內容站，重點是「逛得完、信得過」）、還是兩者都要（品牌站＋商店/預約）？

| archetype | 載什麼 | baseline 重點 |
|---|---|---|
| **交易型應用** | 下面 5 個類型包（Marketplace / B2B SaaS / 消費型 App / 內部工具 / ERP）住這 | 任務流正確性、後台、權限 |
| **呈現型網站**（新）| 「類型 F：呈現型網站」baseline | **資訊架構廣度**：標準區塊齊不齊、到不到得了、信任頁實不實 |
| **混合** | 兩套都載 | 交易正確 ＋ 呈現完整（如 coffeeshooters：品牌站＋商店＋會員）|

**亮區條件**：確認 archetype ∈ {交易型 / 呈現型 / 混合}（判不出 → 第 1 輪意圖擷取後補判）。

> **為什麼**：原本 5 個類型包**全是「應用/系統」原型**，「品牌/內容/呈現型網站」沒有對應 baseline → 收斂只問後端形狀的維度、產出「後端 90 前端 10」。archetype 是 M0 的姊妹樁——把「靜默假設這是交易型應用」變成顯式選擇。網頁呈現只是其中一格，所以「不能只用網頁當標準」靠這個分叉滿足、非例外。

## 維度清單

### 1. 目標用戶
- 誰是主要使用者？
- 有幾種角色？（例：會員、管理員、教練）
- 用戶的技術能力如何？
- **亮區條件**：明確說出至少一種用戶群體

### 2. 核心功能邊界
- 最核心的 1–3 個功能是什麼？
- 明確不做哪些功能？（防止範圍蔓延的關鍵）
- **亮區條件**：列出具體功能名稱，且有「不做」的確認

### 3. 資料模型
- 系統需要儲存哪些資料？
- 有沒有敏感資料？（個資、金流、健康資料）
- 資料的保存期限？
- **【關鍵分叉】Session 資料 vs 持久資料**：
  - 資料需要跨次保留嗎？（歷史記錄、用戶進度、累積數據）
  - 還是每次使用結束後資料可以消失？（課堂遊戲、一次性活動）
  - 這個分叉決定是否需要帳號系統，影響架構複雜度極大，必須主動問清楚
- **亮區條件**：確認主要資料實體，且明確 Session-only 或 Persistent

### 4. 認證與權限
- 需要登入嗎？
- 不同角色有不同的操作權限嗎？
- 登入方式？（帳密 / Google / LINE / 手機號碼）
- **亮區條件**：確認是否需要登入及角色設計

### 5. 第三方整合
- 需要串接金流嗎？（LINE Pay / 信用卡 / 轉帳）
- 需要串接通知嗎？（LINE Notify / Email / SMS / Push）
- 有現有系統需要對接嗎？（ERP / POS / 預約系統）
- **亮區條件**：確認整合清單（可以是空清單）

### 6. 部署平台
- Web 網頁？手機 App？桌面應用？
- 手機優先還是電腦優先？
- 需要上架 App Store / Google Play 嗎？
- **亮區條件**：確認主要使用裝置與平台

### 7. 效能與規模
- 預計同時使用人數？
- 資料量規模（小型 <1000 筆 / 中型 / 大型）？
- 需要即時性嗎？（即時聊天、直播等）
- **亮區條件**：有大概的規模認知

### 8. 維護與更新
- 誰來維護這個系統？
- 多久需要更新一次內容？
- 需要後台管理介面嗎？
- **亮區條件**：確認維護責任歸屬

### 9. 預算與時程
- 大概的預算範圍？（影響技術選型）
- 希望多久可以上線？
- 是一次性開發還是長期維護？
- **亮區條件**：有大致的限制條件

### 10. 成功指標
- 怎麼判斷這個系統「做好了」？
- 上線後最重要的一件事是什麼？
- 有沒有需要達到的業務目標？
- **亮區條件**：用戶能說出至少一個驗收標準

### 11. 用戶旅程與入口
- 每種角色打開瀏覽器 / App 後看到的**第一個畫面**是什麼？
- 從第一個畫面到完成核心任務，中間經過哪些畫面？每一步靠什麼前進（按鈕、連結、自動跳轉）？
- 多角色產品：不同角色在哪裡分流？（同一入口選身份，還是各自不同的 URL？）
- 跨角色的觸發鏈：一個角色的動作，另一個角色在什麼畫面看到什麼結果？
  （例：老師點「開始」→ 學生畫面收到題目——這類「邊」是模組式分解最容易弄丟的東西）
- **亮區條件**：每個角色都有一條從入口到核心任務完成的完整路徑描述

## 類型擴充維度

基準 11 維是所有軟體共用的。以下產品類型各有額外的高頻死角。
**載入邏輯（ADR-002 後，先看 M0.5 archetype）**：
- archetype=**交易型** → 從用戶第一句話判 A–E 哪一型（判不出 → 第 1 輪後補載入），併入 dark。
- archetype=**呈現型** → 載「類型 F：呈現型網站」baseline。
- archetype=**混合** → A–E 對應型 ＋ 類型 F 兩邊都載。
擴充維度優先級插在「第三方整合」之後、「效能規模」之前。
**（A–E 是「交易型應用」的子型；F 是呈現型——這是純加法，舊 5 型不動。）**

### 類型 A：Marketplace（雙邊平台）

**A-11. 雙邊冷啟動**
- 先有供給方還是需求方？最初 50 個供給從哪來？
- 供給不足時需求方看到什麼？（空市場是平台死因第一名）
- 亮區條件：說得出冷啟動的第一步具體做法

**A-12. 交易與抽成模型**
- 錢流經平台（託管/escrow）還是雙方直接交易？
- 誰付手續費？退款與糾紛誰裁決？
- 亮區條件：確認金流路徑與糾紛處理責任方

**A-13. 信任與評價機制**
- 雙向評價？供給方審核（KYC）？詐騙與跳單防範？
- 亮區條件：確認至少一種信任建立機制

### 類型 B：B2B SaaS（多租戶企業用）

**B-11. 租戶隔離模型**
- 共用資料表加 tenant_id，還是 schema-per-tenant，還是獨立部署？
- 客戶會不會要求「我的資料不能跟別人放一起」？
- 亮區條件：確認隔離等級與資料歸屬

**B-12. 企業身份與組織權限**
- 需要 SSO / SAML / Google Workspace 嗎？
- 組織內有層級嗎（admin 委派、部門隔離、審批鏈）？
- 亮區條件：確認登入方式與組織模型

**B-13. 計費與方案結構**
- per-seat / usage-based / 固定方案？試用期？發票與合約（企業採購流程）？
- 亮區條件：確認計費單位（哪怕是「第一版免費」也算）

### 類型 C：消費型 App（C 端社群/娛樂）

**C-11. 內容審核與安全**
- 有 UGC 嗎？審核是事前、事後還是檢舉制？
- 未成年用戶？封鎖/檢舉/隱私設定？（上架審查會問）
- 亮區條件：確認 UGC 有無與最低審核機制

**C-12. 留存與成長迴路**
- 用戶第二天為什麼回來？通知策略？邀請機制？
- 亮區條件：說得出至少一個留存 hook

**C-13. 變現模式**
- 廣告 / 訂閱 / IAP / 完全免費？（影響架構：廣告 SDK、收據驗證）
- 亮區條件：確認第一版的變現（含「不變現」）

### 類型 D：內部工具（公司內部非對外）

**D-11. 既有系統與身份整合**
- 公司有 AD / Google Workspace / SSO 嗎？工具要不要接？
- 資料來源是哪個既有系統（ERP/CRM/Excel）？單向同步還是雙向？
- 亮區條件：確認身份來源與資料上游

**D-12. 資料治理與稽核**
- 誰能看誰的資料（跨部門隔離）？需要操作紀錄（audit log）嗎？法遵要求？
- 亮區條件：確認可見性規則

**D-13. 導入與舊流程退役**
- 現在這件事怎麼做（Excel？紙本？）？舊資料要不要搬進來？
- 誰負責推動同事改用新工具？
- 亮區條件：確認遷移範圍與推動者

### 類型 E：ERP / 複雜企業系統

**E-11. As-is 流程盤點**
- 現行流程（含例外流程、私下的 Excel 流程）有沒有文件化？
- 哪些流程要照搬、哪些要趁機改造？（照搬爛流程是 ERP 失敗主因）
- 亮區條件：每個目標模組都有對應的現狀描述

**E-12. 主數據歸屬（Master Data）**
- 客戶、料號、會計科目表——每種主數據誰建立、誰修改、誰是唯一真相源？
- 模組間共用主數據的同步規則？
- 亮區條件：列出主數據清單與各自的 owner

**E-13. 遷移與並行上線**
- 舊系統資料要搬多少年份？髒資料誰清？
- 新舊系統並行多久？並行期間以哪邊為準？
- 亮區條件：確認遷移範圍與切換策略

### 類型 F：呈現型網站（品牌 / 作品集 / 內容站；archetype=呈現型 或 混合 時載入）

<!-- ADR-002：補「應用/系統」5 型缺的呈現型 baseline。重點不是任務流，是「資訊架構廣度 + 信任 + 媒體充實」。 -->

呈現型網站的價值不是一條任務漏斗，是「逛得完、信得過」。完備性看三軸：

**F-11. 標準區塊清單（資訊架構廣度）**
- 這類網站該有的區塊在不在：首頁 / 關於我 / 商品或服務 / 內容（文章·作品）/ 聯絡 / 頁尾法務（隱私權·服務條款·Cookie）？
- 每條都問「要不要？放導覽哪？」——可以說「不要」，但那是**顯式決定**不是沉默缺口。
- 亮區條件：每個標準區塊都被「要」或「明確不做」表態過。

**F-12. 可達性對稱律**
- 凡規格宣告會存在的面（尤其任何**後台/admin**），都要有一個**能抵達它的入口元素**（導覽連結/按鈕）。
- 反例（實測）：有後台、但前台沒有任何 button 進得去。
- 亮區條件：每個宣告的面都有宣告的入口。

**F-13. 信任頁是產品不是裝飾**
- 關於我 / 品牌故事 / 聯絡，對品牌·電商站＝把陌生人變客人的**轉換器**，不是有空再補的美化。
- 亮區條件：信任頁有實質內容方向（非空殼），且每頁有 owner。

**三層完備性階梯（每個會有內容/媒體的面都過一遍）：**
- **L1 存在**：這個面有沒有？
- **L2 可達**：到得了嗎（有入口）？
- **L3 充實**：有沒有**該有的媒體槽**（圖/影）、**後台有沒有真填料路徑**（上傳/選圖/儲存，非 mock）、實際填了沒？只過 L1/L2 ＝「結構齊全、料全空」的空殼。
- L1/L2/L3 都進 CLAUDE.md「UI/UX 規格」驗收章節（screen 強制輸出），讓前端完備性變驗收條件、不被末段才掃。

### 對輪次骨架的影響

類型擴充維度啟用時，輪次上限從 7 放寬到 9（Complex System 模式），
handoff 門檻同步調整為 `round >= 6 且 dark <= 3`（ERP 類允許帶著
較多已知未知進入分解，由 goal-decomposer 的 Audit 迴圈接手）。

## 螢幕型 UX 擴充維度（modality=螢幕 且 有顧客可見視覺面 時載入）

<!-- spec-sonar v1.3 — 補 grep 零命中的 UX 真死角（設計系統 / 導覽 SSOT）。三態/可逆性走 SKILL.md Pass 2 鏡頭強制輸出欄位（ui_states / critical_flow_reversibility），不在此升維。a11y / i18n 暫不立樁（單一市場現無需求，待對外/法遵/多語客戶 N≥2 再補）。來源：CS 真實事故 + workflow 覆蓋地圖。 -->

**載入條件**：M0 modality=螢幕 **且** 樁 6 確認有「顧客可見的視覺面」（純 API / 純 batch / 純 CLI 後端豁免）。載入時併入 dark，與旅程入口（樁 11）同梯——架構定義性，定錯整體重做。

### S1. 設計系統與品牌視覺 SSOT
- 你有沒有一套固定的**顏色、字體、按鈕長相**，要整個產品從頭到尾長一樣？還是每頁各自好看就好？
- 之後好幾頁**分批做**時，怎麼保證看起來像「同一個產品」？品牌有沒有明確視覺個性（溫暖手感 / 科技俐落 / 極簡）？**深色模式**要不要？
- **亮區條件**：確認「視覺要不要 SSOT」——指認既有設計系統 / 品牌 token，或明確「第一版先不管一致性」（含深色模式決策）。
- **為什麼是轉折點**：多模組分頭實作沒視覺 SSOT 必漂移；token 架構事後抽＝重刷全站。〔化石：CS `synesthesia-design-system`〕

### S2. 導覽清單 SSOT
- 手機版和電腦版的**導覽選單是同一份清單嗎**？（分兩份的話，新增一頁兩邊都要記得加，最容易漏）
- 產品有幾個**主要去處**？站在任一頁怎麼到別處（上方分頁 / 側邊抽屜 / 底部導覽列）？
- **亮區條件**：確認桌機 / 手機導覽是否同一份來源（SSOT）+ 主要導覽元件型態。
- **為什麼是轉折點**：桌機 / 手機導覽各維護一份 → 漂移 → 某裝置使用者找不到頁。〔化石：CS 2026-06-14 `Navigation.vue` vs `CustomerMobileNav.vue` 漂移，修法抽 `config/siteNav.js` SSOT + parity 測試〕

## 初始化邏輯

```
步驟 -1：判斷 modality（M0，最上游分叉 — 先於產品類型與 11 維）
  問「主要介面 = 螢幕 / 語音 / API / 訊息 / 列印？」
  → 螢幕 → 照常進步驟 0
  → 非螢幕 → bright 記 modality=X；known_risks 加「modality=X：螢幕型 UX 假設不適用、對應樁包待長」；旗標 user；不載入螢幕型 UX 維度

步驟 0：判斷產品類型
  （Marketplace / B2B SaaS / 消費型 App / 內部工具 / ERP / 一般）
  → 命中類型 → 將對應擴充維度併入 dark 清單

步驟 0.5：螢幕型 UX 擴充（modality=螢幕 且 有顧客可見視覺面 時）
  → 將 S1 設計系統 SSOT / S2 導覽 SSOT 併入 dark（與樁 11 旅程同梯、架構定義性）
  → 純 API / 純 batch / 純 CLI 後端豁免

for each 維度 in (11 個基準維度 + 已載入的類型擴充維度):
  if 用戶第一句話已涵蓋此維度:
    加入 bright
  else:
    加入 dark

dark 清單按重要性排序：
優先級高：modality(M0 最上游分叉) > 目標用戶 > 功能邊界 > 用戶旅程與入口 > [S1 設計系統/S2 導覽 SSOT，螢幕型] > 認證權限 > 第三方整合 > [類型擴充維度]
優先級低：效能規模 > 維護更新 > 預算時程 > 成功指標
```

`next_focus` 每輪取 `dark` 清單中優先級最高的未確認項目。
