# 任意模型可執行的格式保證機制

<!-- spec-sonar v1.0 — 核心哲學「Fable 做一次深度設計 → 固化成結構 → 任何模型可執行」的落地機制 -->

goal 檔不是「寫得清楚的任務說明」，而是一種**消除執行模型自由度**的格式。
以下八條機制共同保證：Haiku 級的模型也能可靠執行 Fable 級模型設計的 goal。

## 八條機制

1. **自包含原則**
   goal 檔內嵌一切所需（契約複本、資料結構、payload 實例），
   不出現「參見 CLAUDE.md」——弱模型不會可靠地跨檔追溯。

2. **裁決前置**
   所有設計決策由強模型在分解期做完，寫進「前置裁決」區並附理由。
   執行模型的工作被壓縮成「按字面執行」——這是弱模型可靠性的根本來源。

3. **介面字面凍結**
   給 JSON 實例而非 schema 描述。弱模型抄實例不會錯，
   「理解 schema 再生成」會錯。

4. **驗證機械化**
   每條驗收附可執行命令。「測試綠了」取代「我覺得做完了」。

5. **BLOCKED 協議取代猜測**
   給弱模型一條合法的「我不知道」出口（固定格式），
   否則它一定用幻覺填補。

   ```
   BLOCKED
   goal: {id} / 驗收項: {V#}
   類型: ambiguity | dependency-mismatch | environment
   問題: <一句話>
   已嘗試: <事實，非辯解>
   需要的決定: <二或三選一，附建議>
   ```

6. **負向驗收**
   「不做」清單轉成可 grep / 可斷言的檢查
   （例：「資料庫不得存在 users 表」「伺服器重啟後用舊 roomCode join 必須失敗」）。

7. **自檢 footer**
   交付前強制逐項勾選並貼證據，把「宣稱完成」變成「展示完成」。

8. **冷啟動測試（合格判準）**
   把 goal 檔單獨丟給目標最弱模型，問四問——
   做什麼／怎麼驗證／不能做什麼／卡住怎麼辦。
   答不全 = 檔案不合格，**回 decomposer 重寫，而不是換更強的模型**。

## goal 檔五元素結構

每個 goal 檔依以下順序組織（完整範例：examples/mathdefense/goals/G4a.md）：

| 區塊 | 內容 |
|------|------|
| 標頭 | id / tier / depends_on / blocks / contract 版本 |
| 1. 目標成果（Outcome） | 完成的可觀察定義 |
| 2. 驗證方式（Verification） | 每項驗收附機械驗證命令 |
| 3. 約束（Constraints） | 技術棧 + 凍結介面複本 + 前置裁決 + 明確不做複本 |
| 4. 迭代政策（Iteration policy） | 修復嘗試上限、即時驗證、禁止的繞路 |
| 5. 錯誤處理（Error handling） | BLOCKED 協議 |
| 6. 完成自檢 | 逐項勾選 + 貼證據 |
| 附錄 | 契約相關段落全文複本（使檔案自包含） |
