## Q1：能直接從產出的 CLAUDE.md 開工嗎？

**結論：可以開工模組一、二，但模組三、四會在第一天就停下來猜。**

### 猜測點 1：單位數值真空 + 向日葵歸屬矛盾（模組三）

`Unit` 結構定義了 `hp / attackPower / range / speed`，但三種植物、三種殭屍的
實際數值完全沒給。更嚴重的是規格內部矛盾：向日葵「被動產生陽光（每 10s +1）」，
但資源系統明確是「個人賺、個人花」——向日葵產的陽光歸**種它的學生**還是**全隊**？
兩種答案導出完全不同的遊戲設計。

### 猜測點 2：殭屍移動模型未定（模組三 × 模組四交界）

連續 pixel 移動還是格子跳躍？`GridCell.occupant` 是單一 Unit，移動中的殭屍佔哪格？
兩隻殭屍能否同格？這決定 `zombie:moved` 廣播協議的 payload 形狀、碰撞判定、
以及勝負判定「x ≤ 0」的語意（9 欄地圖若索引 0–8，x=0 是植物隊最左欄，
殭屍到 x=0 是「站上底線」還是「越過底線」？）。

### 猜測點 3：斷線重連的身份識別是三方矛盾

驗收要求「斷線重連後地圖狀態自動恢復」，但 `Player` 以 `socketId` 為 key、
重連後 socketId 必然改變、且「不做帳號系統」在禁區。
重連要找回 `resources / correctCount`，必然需要跨連線識別憑證——規格沒給機制。

### 附帶發現：spec-sonar 自身的 schema 漂移

實際產出的 STATE_FINAL.json 偏離 output-templates.md 定義：
- `user_tech_level: "none+partner"` 不在四值枚舉內
- `tech_stack`、`known_risks` 是 ad-hoc 欄位，範本裡沒有

→ 已在 v1.1 升版收編（見 output-templates.md 檔案三）。

## Q2：「明確不做」清單的繞過情境與防彈

### 四種真實會發生的繞過

1. **驗收條件隱性要求禁區功能**：重連問題即實例——執行模型會造一個
   `reconnectToken` 然後說服自己「這不是帳號系統」。
2. **可用性理由侵蝕**：「報告可截圖列印」+ 重新整理就消失 → 想加 localStorage
   ——算不算「跨 session 歷史記錄」？
3. **籠統詞彙的邊界裁量**：「不做排行榜」 vs 「答錯率最高的 3 道題」排序展示。
4. **最佳實踐慣性**：logging、rate limiting、錯誤監控——不在清單也不在禁區。

### 防彈四招

1. **每條「不做」附動機與偽裝形態**：
   「不做帳號系統（動機：降低門檻、保護學生個資）。
   允許：roomCode-scoped 一次性 session token。
   禁止：email/密碼/任何跨房間或跨日識別。」
2. **負向驗收條件（可測試化）**：「資料庫不得存在 users 表」。
3. **衝突上報協議**：驗收條件似乎需要禁區功能時，凍結該項、輸出 CONFLICT、
   等用戶裁決——把繞過從「自由裁量」改成「協議違規」。
4. **handoff 前交叉檢查**：不做清單 × 全部驗收條件配對檢查
   （即 Audit Mode 的 A4 步驟）。

## mathdefense 六模組的模型層級分配（Q8）

| 模組 | 層級 | 理由 |
|------|------|------|
| 一 房間系統 | sonnet | REST + token + QR 是成熟 pattern，但 token 安全設計有自由度 |
| 二 題庫答題 | sonnet | Supabase 整合 + 個人題目佇列有局部決策（去重策略、題庫耗盡行為） |
| 三 遊戲引擎 | **opus** | 數值平衡與移動/碰撞模型規格未給，要發明；是 C2 數值表契約的生產者 |
| 四 即時同步 | **opus** | authoritative server、重連、延遲——架構核心，錯誤擴散全局 |
| 五 老師控制台 | **haiku**（條件式） | 純數據面板，C1 凍結 + G4a 完成後零設計決策；典型葉節點 |
| 六 勝負與報告 | sonnet | 判定邏輯簡單但接三個上游；報告頁與 print CSS 模板化 |
